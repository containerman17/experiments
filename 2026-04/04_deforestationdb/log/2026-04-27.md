## ~14:00 — design session

Long architecture conversation covering:
- **Log indexes**: Erigon E3 has two indexes (`LogAddrIdx`, `LogTopicIdx`),
  both posting TxNums. Topic index is **position-agnostic** — one index
  serves topics 0-3, position resolved at receipt-read time. Reth has
  zero log indexes; relies on per-block `logsBloom` from headers, which
  saturates on modern blocks (~70% FP rate at 1500 inserts in 2048 bits).
  Verified against source.
- **DB layout**: Subnet-evm has a `UseStandaloneDatabase` config flag
  (default true for new chains, false for existing). C-Chain (coreth) is
  always single-DB. ProposerVM always lives in avalanchego's chain DB
  regardless of inner VM choice. Forcing `UseStandaloneDatabase=false`
  unifies everything in one Pebble env.
- **Compression**: Erigon uses custom patricia+huffman (`db/seg/`),
  row-oriented. Reth uses columnar (nippy-jar) with stock LZ4 default,
  optional zstd-with-dict per column. Neither aggressively combines
  columnar + trained zstd — that's the AI-era multi-agent design pass
  opportunity.
- **TxNum vs BlockNum keying**: TxNum aligns with per-tx receipt storage
  (no read-amplification per block hit), modest ~1.5-2x size cost.
- **Bloom limitations**: bloom-only model (geth/reth) is O(range_size),
  fine for live RPC but breaks at archive scale. Index-based (Erigon) is
  O(result_size).

## ~21:00 — three parallel investigations to close the design space

Dispatched three Explore agents:
1. RootStore barrier semantics + Firewood design intent.
2. Firewood production usage in coreth/avalanchego.
3. Pebble fsync cost + creative durability approaches.

Findings:
- **Option C (RootStore as barrier) confirmed UNSAFE.** RootStore is a
  lookup cache; its own comment disclaims OS-crash safety. Rejected.
- **Production reality**: avalanchego/coreth use Firewood at
  `deferred_persistence_commit_count=1`, RootStore disabled in pruning
  mode (the C-Chain default). They accept residual asymmetry risk
  without explicit fsync coordination. Filed
  `wiki/firewood-production-usage.md`.
- **Pebble fsync cost**: "weeks" was overstated. Realistic: 50µs-1ms
  per fsync on commodity NVMe → ~1-23 hours total over 82M blocks.
  Pebble's automatic group-commit amortizes concurrent fsyncs (free
  8-100× speedup). Steady-state at 1 block/sec is trivial.

**Recommendation locked in**: Option B (per-block Pebble Sync=true).
Causal-ordering argument proves Firewood-ahead is impossible. No
upstream patch needed. Firewood loose-durability is fine because it's
always behind Pebble by construction.

Refactored idea file with the recommendation, failure-mode table, and
recovery procedure. Should harden into `decisions.md` after first
end-to-end test.
